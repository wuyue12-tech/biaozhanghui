# 项目总结

本项目已经进入可复用阶段。后续制作同类表彰会时，主线是沿用网页框架，替换本届数据和素材，保留过程资料的归档边界。

## 一、本项目最终目录结构

```text
.
├── index.html                 # 网页入口
├── style.css                  # 舞台视觉、动画、版式
├── script.js                  # 节目类型渲染与交互逻辑
├── data.js                    # 本届活动流程、奖项、名单、节目数据
├── game-data.js               # 游戏题库数据
├── netlify.toml               # Netlify 发布配置
├── AGENTS.md                  # 项目协作规则
├── .gitignore                 # 本地缓存与依赖排除规则
├── audio/                     # 网页实际使用的音频
├── fonts/                     # 网页实际使用的字体
├── images/                    # 网页实际使用的图片
├── video/                     # 网页实际使用的视频
├── tools/                     # 可复用制作工具
├── docs/                      # 方案、流程、标准、交接文档
├── deliverables/              # 现场交付成品
├── materials/                 # 每届原始素材与源数据
└── archive/                   # 过程文件、临时截图、一次性脚本、缓存
```

当前网页由根目录的 `index.html` 直接加载 `style.css`、`data.js`、`game-data.js`、`script.js`。`netlify.toml` 配置为发布项目根目录，因此网页运行文件和它们引用的 `audio/`、`fonts/`、`images/`、`video/` 路径应保持稳定。

## 二、下次制作表彰会时需要修改哪些文件

下次优先修改 `data.js`。这里包含活动标题、日期、主持人、流程顺序、奖项名称、获奖名单、节目名称、表演者、发言人、视频路径、图片路径、谢幕文案。

需要替换的网页素材放在固定位置：

- `images/seat-map.png`：座位表。
- `images/signwall-preview.png`：签名墙或合影背景预览图。
- `images/student-speech/`：学生发言图片。
- `images/game/`：互动游戏图片。
- `video/warmup.mp4`：暖场视频。
- `video/` 中的节目视频：名称要与 `data.js` 里的路径一致。
- `audio/`：背景音乐和音效；如果替换音频，同步更新 `audio/SOURCES.md`。

如果更换互动游戏图片，先把图片放进 `images/game/`，再运行 `tools/generate_game_data.mjs` 生成新的 `game-data.js`。当前工具适合“猜教室、猜老师物品、猜老师照片”这三类游戏。

原始资料保留在 `materials/`：

- `materials/award-data/`：获奖名单原始表格。
- `materials/source-images/`：游戏照片原始素材。
- `materials/source-slides/`：学生发言原始图片。
- `materials/source-video/`：节目或暖场源视频。
- `materials/source-assets/`：签名墙、背景图、字体等制作源素材。

现场交付物放在 `deliverables/`。签名墙最终版、PPT 成品、预览图和历史版本不放回根目录。

## 三、哪些内容可以直接复用

可以直接复用的内容是框架层：

- `index.html`：全屏舞台入口。
- `script.js`：节目类型渲染器，支持座位表、暖场、开场、章节转场、奖项、表演、发言、图片发言、互动游戏、合影、谢幕。
- `style.css`：星空舞台视觉、名单排版、视频展示、游戏揭晓、菜单和进度条样式。
- `netlify.toml`：当前静态发布方式。
- `fonts/`：当前中文展示字体。
- `tools/generate_game_data.mjs`：游戏题库生成工具。
- `audio/SOURCES.md`：音频授权记录格式。
- `docs/ceremony/入境仪式成功标准.md`：八升九入境仪式类活动的主线校准标准。

可以复用的工作方式是三层分离：

```text
Framework：网页框架、渲染逻辑、视觉系统、部署配置
Data：活动流程、奖项名单、节目安排、游戏题库
Assets：图片、音频、视频、字体、签名墙、PPT、源素材
```

下次制作时，先判断变的是 Data 还是 Assets。只有新增节目类型、改变交互方式、改变整体视觉系统时，才进入 Framework。

## 四、哪些内容建议以后进一步抽象

优先抽象 `data.js`。建议未来拆成：

```text
event.json      # 活动标题、日期、主持人、基础素材路径
program.json    # 流程顺序和节目安排
awards.json     # 奖项、说明、获奖名单
games.json      # 游戏组别、题目、答案、展示模式
assets.json     # 图片、音频、视频、字体路径
```

第二个抽象是节目类型说明表。每种 `type` 应写清楚必填字段，例如：

- `award`：`category`、`title`、`subtitle`、`names` 或 `groups`。
- `performance`：`title`、`performer`、`video`。
- `speech-slides`：`title`、`speaker`、`images`。
- `game`：由 `game-data.js` 或未来的 `games.json` 提供题库。

第三个抽象是名单转换工具。当前获奖名单已经结构化，但仍需要人工写入 `data.js`。以后可以从 Excel 自动生成奖项数据，减少错名、漏名、顺序错的问题。

第四个抽象是游戏配置。当前 `tools/generate_game_data.mjs` 写死了三类游戏和题目顺序。以后可以改成读取配置文件，让题型、题目顺序、答案提取规则都可配置。

第五个抽象是发布前检查清单。建议以后形成自动检查：

- `data.js` 和 `game-data.js` 语法可读。
- 所有图片、音频、视频路径存在。
- 获奖名单人数与原始表格一致。
- 视频能播放，音频来源已记录。
- Netlify 发布路径仍为项目根目录。

## 五、本项目有哪些经验值得保留

第一，根目录只保留入口文件。网页运行、素材、交付物、源资料、过程文件各自归位，后续维护成本会明显降低。

第二，框架和数据分开。`script.js` 负责机制，`data.js` 负责本届内容。下次主要改数据，不轻易动框架。

第三，节目先变成数据，再进入视觉。奖项、节目、游戏、发言都用统一结构表达，页面效果由渲染器自动生成。

第四，现场素材保留两套：网页实际使用的一套放在 `images/`、`audio/`、`video/`；原始素材放在 `materials/`。这样既能保证网页稳定，也能追溯来源。

第五，过程文件不丢，但退出主线。临时截图、旧版本、一次性脚本进入 `archive/`，不干扰下次判断。

第六，签名墙、PPT、流程图这类现场成品应进入 `deliverables/` 或 `docs/`，与网页代码保持距离。

第七，八升九入境仪式相关内容继续以 `docs/ceremony/入境仪式成功标准.md` 为校准标准。方案和网页表达都应服务“学生共创，团队闯关，家长见证，导师认证”这条主线。

第八，音频和外部素材要保留来源说明。`audio/SOURCES.md` 的格式可以直接复用，避免后续无法判断素材是否可继续使用。
